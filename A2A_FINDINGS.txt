CRITICAL FINDING:
═════════════════════════════════════════════════════════════════════════════════
Three completely SEPARATE, UNCONNECTED agent systems exist in parallel:

1. SYSTEM A: CHAT ORCHESTRATION (/src/stock_datasource/agents/)
   - OrchestratorAgent → LLM-based intent routing
   - Routes to: MarketAgent, ReportAgent, HKReportAgent, etc.
   - Events: type="thinking", debug_type="classification|routing|agent_start|..."
   - SSE Endpoint: /chat/stream

2. SYSTEM B: ORCHESTRATION PIPELINES (/src/stock_datasource/services/orchestration_*.py)
   - NEW in commit 032d83f (feat: Agent中心 — 可配置化Agent平台)
   - DAG-based execution: topological sort → sequential agent nodes
   - AgentConfig stored in ClickHouse (configurable agents)
   - Events: type="node_start|node_end|node_error"
   - SSE Endpoint: /api/orchestrations/{id}/execute

3. SYSTEM C: ARENA DISCUSSIONS (/src/stock_datasource/arena/)
   - Multi-agent debate/collaboration/review mode
   - ArenaAgentBase agents
   - Publishes: DecisionSummary with signal (BUY/SELL/HOLD) + confidence
   - SSE Endpoint: /api/arena/{arena_id}/thinking-stream
   - ISOLATED: Only runs if explicitly created


WHY CHAT "决策" (DECISIONS) SIDEBAR IS EMPTY:
═════════════════════════════════════════════════════════════════════════════════

Problem:
--------
The chat page's "决策" sidebar is designed to show agent discussions and decision
signals (BUY/SELL/HOLD voting), but it always appears EMPTY.

Root Cause:
-----------
1. Chat system (System A) does NOT invoke Arena (System C)
2. Chat system emits "debug events" (tool calls, thinking), NOT decision signals
3. Arena system is completely isolated - only runs if explicitly created
4. NO bridge exists to cross-publish events between systems
5. Chat session has NO associated Arena instance

Evidence:
---------
- /src/stock_datasource/modules/chat/router.py::_stream_response()
  Line 249: orchestrator.execute_stream(content, context)
  └─ Only calls OrchestratorAgent, not Arena
  
- /src/stock_datasource/services/execution_planner.py
  AGENT_HANDOFF_MAP only defines handoffs WITHIN specialized agents:
    MarketAgent → ReportAgent, HKReportAgent, BacktestAgent
    ScreenerAgent → MarketAgent, ReportAgent
  └─ NO handoffs to Arena agents or Orchestration agents

- DecisionSummary only generated in:
  /src/stock_datasource/arena/discussion_orchestrator.py::_generate_decision_summary()
  └─ This is ONLY called in Arena, not in Chat flow

HO = Handoffs (within-system only)


MISSING A2A PROTOCOL:
═════════════════════════════════════════════════════════════════════════════════

1. ❌ NO AGENT DISCOVERY SERVICE
   - Specialized agents hardcoded in /src/stock_datasource/agents/__init__.py
   - Configurable agents in ClickHouse agent_configs table
   - No registry to find agents across systems

2. ❌ NO HANDOFF MECHANISM BETWEEN SYSTEMS
   - Within System A: MarketAgent → ReportAgent (works)
   - Between Systems: NO delegation (Chat → Orchestration, Chat → Arena)

3. ❌ NO SHARED MESSAGE FORMAT
   System A event:  {"type": "thinking", "debug_type": "agent_start", ...}
   System B event:  {"type": "node_start", "node_type": "agent", ...}
   System C event:  {"type": "message", "message_type": "thinking", ...}

4. ❌ NO UNIFIED EVENT BUS
   System A: events → memory + chat_messages table
   System B: events → memory + orchestration_executions table
   System C: events → ThinkingStreamProcessor + thinking_messages table
   └─ No cross-system pub/sub

5. ❌ NO DECISION SIGNAL BRIDGE
   Only System C computes decision signals (DecisionSummary)
   Chat cannot access them (separate Arena instance)

RECOMMENDED SOLUTIONS:
═════════════════════════════════════════════════════════════════════════════════

Option 1: EMBED ARENA IN CHAT FLOW (Comprehensive)
  - When user sends chat message, also run parallel Arena discussion
  - Capture decision signal from Arena → display in sidebar
  - Cons: +2-3x latency, +2x compute

Option 2: INTEGRATE ORCHESTRATION INTO CHAT (Moderate)
  - Auto-generate pipeline from chat intent
  - Execute pipeline for multi-step analysis
  - Cons: Different UI paradigm, frontend refactor

Option 3: CREATE A2A MESSAGE BUS (Architectural)
  - Unified event bus (Redis pub/sub or ClickHouse log)
  - Each system publishes to bus
  - Frontend subscribes to multiple topics
  - Cons: Most complex, biggest refactor


CONCLUSION:
═════════════════════════════════════════════════════════════════════════════════

The stock_datasource project contains THREE ISOLATED AGENT SYSTEMS that do not
communicate. The chat system (System A) does NOT invoke the Arena system (System C),
which is why the "决策" sidebar remains empty.

To enable "决策" in chat:
  1. Choose resolution option above
  2. Implement integration layer
  3. Update frontend to consume decision signals

No A2A protocol currently exists between the systems.
